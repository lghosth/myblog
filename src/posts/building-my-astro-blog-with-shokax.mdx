从很早开始，我就一直觉得应该有一个长期维护的个人博客。相比把零散笔记放在聊天记录、收藏夹或临时文档里，博客更像是一个可以持续沉淀的个人知识库：它不一定每天更新，但每次更新都应该能留下清晰的上下文。

这篇文章是本站的第一篇建站记录，主要整理从技术选型、主题选择，到本地初始化、部署上线和后续配置规划的过程。它既是一次搭建记录，也是一份给之后自己的维护手册。

<Note type="info" title="本文目标">
  本文不会追求把每个命令都写成完整教程，而是重点记录为什么选择当前方案、如何完成初始化，以及后续维护时需要注意哪些细节。
</Note>

## 为什么重新搭建博客

这次搭建博客，我对它的定位比较明确：

- 用来记录技术学习、项目实践和日常折腾。
- 页面要足够轻量，访问速度不能拖后腿。
- 写作流程要简单，最好直接使用 Markdown 或 MDX。
- 部署和维护成本要低，不希望为了博客本身花费太多精力。
- 主题要有一定完成度，但又不能把二次定制变得很困难。

换句话说，我希望它是一个“能长期用下去”的内容站，而不是一个只在搭建当天看起来很酷、之后却很难维护的玩具项目。

## 博客框架技术选型

个人博客常见方案其实很多，大致可以分为两类：动态博客和静态博客。

动态博客通常会有后台、数据库和服务端运行环境，例如 WordPress 这类系统。它的优势是管理方便、生态成熟，但对个人博客来说，服务器维护、安全更新、数据库备份等成本也会随之而来。

静态博客则更适合偏内容展示的场景。文章在构建阶段生成 HTML，部署时只需要托管静态资源，访问速度快，安全风险低，也更容易迁移。

| 方案 | 优点 | 我个人的顾虑 |
| --- | --- | --- |
| Hexo | 生态成熟，主题丰富，上手简单 | 技术栈相对传统，后续定制不够顺手 |
| Hugo | 构建速度非常快，适合大型静态站 | Go Template 对我来说维护成本偏高 |
| VitePress | 文档体验很好，配置简洁 | 更偏文档站，博客能力需要额外补齐 |
| Next.js / Nuxt | 能力完整，生态强大 | 对纯博客来说偏重，部署心智更复杂 |
| Astro | 内容站友好，默认静态优先，组件岛灵活 | 生态规模不如一些老牌博客框架 |

最终我选择了 Astro。原因很简单：它足够现代，又刚好适合内容站。

Astro 的几个特点非常契合个人博客：

- 默认以静态内容为核心，适合文章、归档、标签这类页面。
- 支持 Markdown / MDX，写作体验比较自然。
- 可以按需引入交互组件，不必让整站都变成前端应用。
- 对性能和 SEO 比较友好，生成出来的页面足够轻。
- 和现代前端生态结合顺畅，后续扩展空间比较大。

## 主题选型：为什么是 ShokaX

确定框架之后，下一步就是主题选择。对我来说，主题并不只是“外观模板”，它还决定了之后写作、维护和扩展的基础体验。

我选择主题时主要看几个方面：

- 默认视觉效果是否符合个人审美。
- 是否支持文章目录、标签、分类、归档、搜索等博客基础能力。
- 配置是否集中，后续修改是否方便。
- 移动端体验是否过得去。
- 技术栈是否符合自己的维护习惯。
- 是否有继续二次开发的空间。

ShokaX 的完成度比较高，默认风格也比较接近我想要的感觉：有一定视觉设计，但不会过度复杂；功能比较齐全，但大部分能力又能通过配置开关控制。

<Note type="success" title="选择 ShokaX 的核心原因">
  它让我可以先把博客跑起来，尽快进入写作状态；同时又保留了足够的定制空间，方便之后慢慢调整成自己的样子。
</Note>

目前这个博客基于 Astro Blog ShokaX，整体技术栈大致如下：

- Astro：站点框架与静态生成。
- Svelte 5：局部交互组件。
- UnoCSS：样式组织与工具类。
- Pagefind：站内搜索。
- Bun：包管理和脚本运行。
- Markdown / MDX：文章写作格式。

## 本地初始化与运行

项目初始化的第一步是准备本地环境。因为当前项目使用 Bun 作为包管理器，所以本地需要先安装 Bun，并尽量保持版本和项目声明一致。

常用流程大致如下：

```bash
bun install
bun run dev
```

启动开发服务后，就可以在本地浏览器预览站点效果。之后每次修改主题配置、文章内容或样式，都可以通过本地服务快速确认效果。

常用脚本可以简单分为几类：

| 命令 | 用途 |
| --- | --- |
| `bun run dev` | 启动本地开发服务 |
| `bun run build` | 构建生产版本 |
| `bun run preview` | 预览构建后的站点 |
| `bun run format` | 格式化代码和内容 |
| `bun run lint` | 运行代码检查 |
| `bun run check` | 运行 Astro / Svelte 类型检查 |

<Note type="warning" title="初始化时需要注意">
  本地能启动不代表线上一定能构建成功。部署平台的 Bun / Node 环境、构建命令、输出目录都需要和项目保持一致。
</Note>

## 主题基础配置

ShokaX 的站点配置集中在主题配置文件中。对个人博客来说，最开始不需要一次性改完所有配置，优先处理会直接影响站点展示的部分即可。

我把初始化配置分成了几个阶段。

### 站点基础信息

首先是博客名称、站点标题、站点描述、语言、导航菜单等基础信息。这些内容会影响首页展示、浏览器标题、SEO 信息和站点整体识别度。

这部分配置建议尽早确定，因为它是整个博客的“门面”。

### 作者与侧边栏

作者信息包括昵称、头像、简介和社交链接。侧边栏是访客进入博客后很容易注意到的位置，所以这里的文字不需要很长，但应该能说明这个博客大概会写什么。

我的目标是让它保持简洁：告诉读者我是谁、这里会记录什么、可以从哪里联系到我。

### 导航与页面

博客常见导航包括：

- 首页
- 归档
- 分类
- 标签
- 关于
- 友链

这些页面基本覆盖了内容浏览、信息组织和站点介绍。后续如果有新的长期栏目，也可以再补充到导航中。

### 文章 frontmatter

文章内容主要放在 `src/posts` 目录中，每篇文章通过 frontmatter 描述基础信息。例如：

```yaml
title: 文章标题
date: 2026-04-24
updated: 2026-04-24
description: 文章摘要
tags:
  - Astro
  - 博客搭建
categories:
  - 建站记录
draft: true
```

其中 `draft` 很适合用来区分草稿和正式发布内容。像本文这种初稿阶段，我会先保持草稿状态，等内容补完、链接检查和本地构建都没问题之后再发布。

### 友链与关于页

博客搭建完成后，我还会继续完善关于页和友链页。关于页负责介绍博客和作者，友链页则用于连接常读站点或互相交换链接的朋友。

友链配置通常需要包含站点地址、名称、描述、作者和头像。头像这类静态资源可以放在 `public` 目录下，然后使用以 `/` 开头的路径引用。

<Collapse title="我的初始配置优先级" open>
  先让站点能稳定访问，再完善作者信息、导航、关于页和友链页；最后再考虑评论、音乐播放器、动效和更细节的视觉定制。
</Collapse>

## 部署上线流程

静态博客最大的好处之一，就是部署流程足够简单。理论上，只要平台能执行构建命令并托管 `dist` 目录，就可以完成上线。

### 大致部署流程

我这次的部署思路可以简单拆成三步：代码托管、页面部署和域名解析。

第一步是把博客项目推送到 GitHub。这样做的好处是，后续每次写文章或修改配置，都可以通过 Git 记录变更；部署平台也能直接监听仓库更新。

第二步是在 GitHub Pages 或其他静态站平台中配置构建流程。如果使用 GitHub Pages，通常需要注意几个关键点：

- 选择正确的部署分支，例如 `main`。
- 配置构建命令，例如 `bun run build`。
- 确认最终输出目录是 `dist`。
- 如果使用 GitHub Actions，需要确保运行环境能安装并使用 Bun。

第三步是绑定自己的域名。域名可以在注册商处购买，然后把 DNS 托管到 Cloudflare，后续解析、HTTPS 和 CDN 相关能力都比较方便统一管理。

以 Cloudflare DNS 为例，大致会配置两类记录：

```txt
CNAME  blog     your-github-pages-domain.github.io
CNAME  www      your-github-pages-domain.github.io
```

如果是根域名，也可能会用到 `A` 记录或 Cloudflare 的 CNAME Flattening，具体取决于部署平台提供的接入方式。配置完成后，再到 GitHub Pages 或对应部署平台里填写自定义域名，并等待 HTTPS 证书签发。

<Note type="warning" title="域名解析不是立即生效">
  DNS 修改后通常需要等待一段时间才会在全球范围内完全生效。刚配置完时如果访问异常，不一定是部署失败，也可能只是解析还没刷新。
  P.S. 这也是为什么部署前最好先确认站点能在默认域名下正常访问，避免上线后还要调试部署问题。我的解析就是花了将近两个小时才解析上，虽然已经传播了DNS记录，但是不知道Cloudflare显示还未生效。
</Note>

部署时我主要关注这几个点：

- 仓库代码是否已经推送到 GitHub。
- 部署平台是否连接了正确的仓库和分支。
- 构建命令是否设置为 `bun run build`。
- 输出目录是否设置为 `dist`。
- 自定义域名和 HTTPS 是否配置完成。
- 线上页面的内部链接、图片和搜索是否正常。

部署完成后，不要只看首页是否能打开，还要检查文章页、归档页、标签页、分类页和静态资源路径。尤其是博客开启尾斜杠规则时，内部链接最好保持统一，避免出现重复页面或跳转异常。

## 部署过程中的实际改动

从第一次把站点跑起来，到真正准备上线，中间其实做了不少小修小补。它们单独看都不复杂，但很适合记录下来，因为这些细节往往就是主题初始化和部署过程中最容易被忽略的地方。

### 站点主题配置从默认值改成自己的内容

一开始我先把主题的默认配置逐步替换成自己的站点信息，例如导航、首页标题、页脚、右侧卡片和友链入口。这里最重要的不是一次性把所有功能打开，而是先确认哪些页面对当前博客真正有用。

例如导航配置里，我保留了首页、关于、文章、动态、统计和友链几个入口，并且注意内部链接都保留尾斜杠：

```ts
nav: [
  { href: "/", text: "首页", icon: "i-ri-home-line" },
  { href: "/about/", text: "关于", icon: "i-ri-user-3-line" },
  { href: "/friends/", text: "友链", icon: "i-ri-link" },
];
```

首页配置也从默认展示示例分类，调整成更适合空站初始化的方式：

```ts
home: {
  selectedCategories: [],
  pageSize: 5,
  title: {
    behavior: "custom",
    customTitle: "Charol的博客",
  },
},
```

这样做的好处是：博客刚开始内容不多时，首页不会被示例分类或无意义模块占满，后续等文章数量增加后再打开更多展示区域也不迟。

### 移动端侧边栏滚动问题

部署前测试移动端时，我发现侧边栏打开后滚动体验不太理想：页面主体和侧边栏可能同时滚动，容易产生穿透感。这个问题在桌面端不明显，但在手机上会影响使用体验。

修复思路是：当移动端侧边栏打开时，锁定 `body` 的滚动；侧边栏自身使用独立滚动区域。核心逻辑类似这样：

```ts
const shouldLock = isSidebarOpen && window.innerWidth < 1024;

if (shouldLock && !bodyScrollLocked) {
  previousBodyOverflow = document.body.style.overflow;
  document.body.style.overflow = "hidden";
  bodyScrollLocked = true;
}
```

对应的样式也需要让侧边栏占满移动端视口，并允许自身滚动：

```css
#sidebar {
  top: 0;
  bottom: 0;
  height: 100dvh;
  max-height: 100dvh;
  overflow-y: auto;
  overscroll-behavior: contain;
  -webkit-overflow-scrolling: touch;
}
```

这个改动提醒我：部署前不能只在桌面浏览器看首页，至少要用移动端尺寸检查导航、侧边栏、文章页和长内容滚动。

### MDX 细节：Spoiler 和任务列表样式

因为这个博客会使用 MDX 写文章，我也顺手检查了自定义语法和 GFM 任务列表的显示效果。这里遇到的一个细节是，隐藏文本和任务列表 checkbox 的样式需要和主题正文样式协调，否则会出现显示不自然的问题。

比如 spoiler 的隐藏效果，最终更偏向“默认遮挡，悬停显示”：

```css
.md .spoiler:not(.bulr) {
  background-color: var(--grey-4);
  color: transparent;
  border-radius: 0.125rem;
  cursor: help;
}

.md .spoiler:not(.bulr):hover {
  background-color: transparent;
  color: var(--text-color);
}
```

任务列表则保留 checkbox，而不是直接隐藏：

```css
.md ul li.task-list-item {
  display: flex;
  align-items: flex-start;
  gap: 0.55rem;
}

.md ul li.task-list-item input[type="checkbox"] {
  display: inline-block;
  accent-color: var(--primary-color);
  pointer-events: none;
}
```

这些改动不属于“部署配置”，但属于正式上线前的内容体验检查。毕竟博客最终是给文章服务的，正文样式稳定比动效更重要。

### 图表颜色和懒加载

统计页使用 ECharts 展示文章数据时，也做过一次优化。主要问题是 Canvas 图表对 `var()`、`color-mix()` 这类 CSS 颜色表达式支持并不总是可靠，可能导致颜色回退或显示异常。

处理方式是先在浏览器里把 CSS 颜色解析成实际的 `rgb/rgba` 字符串，再传给 ECharts：

```ts
function resolveCssColor(expression: string): string {
  const probe = document.createElement("span");
  probe.style.color = expression;
  document.body.append(probe);

  const resolved = window.getComputedStyle(probe).color.trim() || expression;
  probe.remove();

  return resolved;
}
```

同时，ECharts 也改成按需动态导入，避免在页面初始阶段就加载较重的图表模块：

```ts
const [core, charts, components, renderers] = await Promise.all([
  import("echarts/core"),
  import("echarts/charts"),
  import("echarts/components"),
  import("echarts/renderers"),
]);
```

这类优化不一定是第一天必须做的，但如果站点有统计页、图表页或较重交互组件，就值得在上线前关注一下。

### 友链页面顺序和头像资源

上线前还补充了第一个友链，并把友链列表移动到页面更靠前的位置。原本页面顺序更偏“规则说明优先”，但实际访问时，用户大概率是先想看友链列表，再决定是否阅读交换规则。

友链配置大致如下：

```ts
friends: {
  title: "友链",
  description: "这里会放一些常读、常用或互相交换的站点。",
  links: [
    {
      url: "https://blog.rimuru.work/",
      title: "Rimuru's Blog",
      desc: "分享 DevOps、AI 工具、Linux 运维及游戏攻略。",
      author: "Rimuru",
      avatar: "/avatar/friend-avatart-rimuru.png",
    },
  ],
},
```

头像资源放在 `public/avatar` 下，引用时直接使用 `/avatar/...`。这也是静态资源路径最容易踩坑的地方之一：`public` 目录里的文件不会再通过 `src/assets` 的导入链处理，而是直接从站点根路径访问。

<Note type="info" title="这部分改动的经验">
  真正上线前的工作，不只是把项目部署成功，还包括移动端体验、正文样式、静态资源路径、配置默认值和页面信息架构这些细节。它们不会决定博客能不能打开，但会明显影响博客是否“像一个已经准备好的站点”。
</Note>

## 踩坑与维护记录

这类项目真正麻烦的地方，往往不是第一次跑起来，而是后续维护。为了减少之后遗忘，我把目前比较重要的注意点记录下来。

### Bun 版本要尽量一致

项目使用 Bun 作为运行时和包管理器。不同版本之间如果依赖解析行为有差异，可能会导致 lockfile 或构建结果变化。因此最好在本地和部署平台都使用接近的 Bun 版本。

### 静态资源路径要统一

放在 `public` 目录下的资源，会以站点根路径暴露。例如 `public/avatar/example.png` 应该通过 `/avatar/example.png` 引用。

如果资源放在 `src/assets` 中，则通常会走 Astro 的资源处理流程，适合需要构建优化的图片。

### 配置改动后及时检查

主题配置往往会影响很多页面。每次修改配置后，最好至少运行：

```bash
bun run format
bun run lint
bun run check
```

如果涉及部署，还应该再跑一次构建，确认生产环境没有问题。

### 不要一次性改太多

主题定制很容易越改越多。我的计划是先保持主线稳定，再逐步调整局部功能。这样即使遇到问题，也比较容易定位是哪个改动引入的。
